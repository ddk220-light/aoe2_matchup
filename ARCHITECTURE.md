## Two Databases

The system uses **two SQLite databases** with different purposes:

| Database | File | Purpose | Generated by |
|----------|------|---------|-------------|
| **Main DB** | `webapp/aoe2_units.db` | Webapp runtime: unit stats per civ, comments, verifications | `generate_database.py` |
| **Reference DB** | `webapp/aoe2_reference.db` | Audit trail: base/final stats, tech chains, stat progressions, special effects, projectiles | `database_creation/generate_reference.py` |

The main DB has a flat `unit_stats` table with 22+ combat property columns used by the battle simulator and unit browser. The reference DB has 5 normalized tables (`ref_units`, `ref_techs_applied`, `ref_stat_chain`, `ref_special_effects`, `ref_projectiles`) used by the analysis page and matchup advisor.

See [DESIGN.md](DESIGN.md) for full schema documentation.

## Pre-computed Data

| File | Purpose | Generated by |
|------|---------|-------------|
| `webapp/battle_scores.json` | Round-robin + benchmark battle scores per unit line | `webapp/compute_battle_scores.py` |
| `webapp/matchup_votes.jsonl` | User combo votes (append-only log) | Matchup advisor vote endpoint |

`battle_scores.json` is loaded into memory at Flask startup. It contains:
- **Round-robin scores**: For each unit line × age, every civ's unit is simulated against every other civ's unit (30v30 + 3000-resource). Results: score_30v30, score_3k, score_5k.
- **Benchmark scores**: Each unit simulated against reference opponents (Champion, Paladin, Arbalester).

## Product Features

### 1. Unit Browser (`/units`)

**Template:** `index.html` | **API:** `/api/unit/<age>/<slug>`

Browse unit types across all civilizations. Select an age (Feudal/Castle/Imperial) and a unit type to see a comparison table showing each civ's version with stats, bonuses, and combat efficiency.

**Data flow:**
1. Page loads → fetches `/api/units` for unit list grouped by age
2. User selects unit → fetches `/api/unit/<age>/<slug>`
3. Returns all civs' stats for that unit, highlighting values that differ from baseline
4. Shows missing civs (civs that don't have that unit)

**Data source:** Main DB (`unit_stats` table)

### 2. Civilization Browser (`/civ`, `/civ/<name>`)

**Templates:** `civ_select.html`, `civ_detail.html` | **API:** `/api/civ/<name>`

Grid of 13 civilization cards. Click a civ to see all its available units organized by age, with stats and civ bonuses.

**Data flow:**
1. `/civ` shows grid of all civs
2. Click a civ → `/civ/<name>` loads `civ_detail.html`
3. Fetches `/api/civ/<name>` → returns all units grouped by age
4. Also fetches `/api/ref/civ/<name>` for reference data (tech chains, stat progressions)

**Data source:** Main DB + Reference DB

### 3. Battle Simulator (`/simulate`)

**Template:** `simulate.html` | **API:** `/api/combat-unit/<civ>/<slug>`, `/api/civ-units/<civ>/<age>`

Interactive 1v1 unit battle simulator with full 2D canvas visualization.

**Data flow:**
1. User selects two civs, age, and units
2. Frontend fetches `/api/civ-units/<civ>/<age>` for available unit lists
3. Fetches `/api/combat-unit/<civ>/<slug>` for each selected unit
4. **Frontend runs its own JavaScript simulation** with 2D positions, movement, projectile animations
5. Backend simulation (`simulation.py`) is used only by the Matchup Advisor — the battle sim page runs entirely client-side

**Architecture note:** The frontend simulation (`simulate.html` JavaScript with `BattleUnit` class) has full 2D positions, avoidance, and projectile rendering. The backend simulation (`simulation.py`) is damage-only with no positions — it's used for batch scoring where speed matters (~1.3ms/sim).

**Data source:** Main DB (`unit_stats` table)

### 4. Matchup Advisor (`/matchup-advisor`)

**Template:** `matchup_advisor.html` | **APIs:** `/api/matchup-advisor/analysis/<c1>/<c2>`, `/api/matchup-advisor/army/<c1>/<c2>`

The most complex feature. Compares two civilizations across Castle and Imperial age using a 4-phase analysis:

**Phase 1 — Mobile Dominance:** Identifies which civ controls the map with fast units (speed > 1.3). Runs all mobile unit pairs through `simulate_battle()` and scores them.

**Phase 2 — Ranged Advantage:** Finds each civ's best ranged unit (attack_range > 1.0, gold cost > 0, non-siege). Determines if either civ has a clear ranged advantage.

**Phase 3 — Army Combos:** Builds up to 4 two-unit army compositions per civ based on:
- Mobile dominance results
- Ranged advantage results
- Best melee/siege support options
- Combo diversity (avoids duplicate pairs, varies primary units)

**Phase 4 — Army Simulation:** Tests all combo pairs with 5 resource splits (70/30 to 30/70) at 5000 resources using `simulate_mixed_battle()`. Finds optimal split per matchup and identifies best combo and counters.

**Progressive loading:**
1. Frontend fires `/api/matchup-advisor/analysis/<c1>/<c2>` (phases 1-3, fast ~1.5s)
2. Renders 3-column analysis + combo suggestions immediately
3. Fires `/api/matchup-advisor/army/<c1>/<c2>` (phase 4, slow ~4s) asynchronously
4. Army results populate when ready; recommended combo is revealed

**Key helper functions in `app.py`:**
| Function | Purpose |
|----------|---------|
| `_categorize_units()` | Splits units into mobile/ranged_gold/siege pools |
| `_run_pair()` | Runs 30v30 + 3000-resource sim, returns 3-point score |
| `_find_clear_winner_and_scores()` | Runs all pairs, finds clear winner |
| `_find_best_counter()` | Finds unit scoring best against target pool |
| `_build_combos_for_civ()` | Builds 4 diverse combo options |
| `_run_army_sims()` | Phase 4 mixed army simulations |
| `_run_matchup_analysis()` | Shared phases 1-3 (used by both endpoints) |

**Data source:** Reference DB → `_build_combat_dict_from_ref()` → `prepare_combat_unit()` → simulation

### 5. Unit Analysis / Audit (`/analysis`)

**Template:** `analysis.html` | **API:** `/api/ref/civ/<name>`, `/api/ref/verify/`

Developer tool for verifying unit stat accuracy. For each civ, shows every unit with:
- Base stats vs final stats
- Complete tech chain with effects
- Stat progression (snapshot after each tech)
- Special combat properties with sources
- Projectile data

Units can be marked as "verified" to track audit progress.

**Data source:** Reference DB (all 5 tables)

### 6. Unit Line Rankings

**API:** `/api/ref/unit-line/<line_slug>`

Compares units within a unit line (e.g., all civs' Knight line) with pre-computed battle scores. Used by the civ detail page to show comparative rankings.

**Data flow:**
1. Request `/api/ref/unit-line/knight` → returns all civs' knights + unique unit substitutes
2. Each entry includes pre-computed scores from `battle_scores.json`
3. Scores: round-robin (30v30, 3k resources, 5k resources) + benchmarks (vs Champion, Paladin, Arbalester)

**Data source:** Reference DB + `battle_scores.json`

### 7. Comment System

**APIs:** `/api/comments`, `/api/simulation-comments` | **Pages:** `/review`, `/simulation-notes`

Two comment systems:
- **Unit comments** (`/api/comments`): Per-cell comments on unit stat tables (unit + civ + column)
- **Simulation comments** (`/api/simulation-comments`): Notes on specific battle matchup results

Both stored in Main DB tables (`comments`, `simulation_comments`).

## Simulation Engine (`webapp/simulation.py`)

The backend simulation is a tick-based damage calculator with no positions or movement.

### Key Functions

| Function | Signature | Purpose |
|----------|-----------|---------|
| `prepare_combat_unit(row)` | `dict → dict` | Parse DB row/dict once, pre-compute derived fields |
| `simulate_battle(u1, u2, resources, ...)` | `→ (winner, rem1, rem2)` | Single unit type vs single unit type |
| `simulate_mixed_battle(team1, team2, ...)` | `→ (winner, rem1, rem2)` | Mixed army (multiple unit types per side) |

### Simulation Flow

```
1. Army Sizing
   resources / unit_cost = count per side

2. Opening Volley (ranged advantage)
   - Ranged vs melee: free shots = closing_time / reload_time
   - Ranged vs ranged: free shots = range_difference / 2

3. Tick Loop (DT=0.1s, max 2500 ticks = 250s)
   For each alive attacker:
     - Check reload timer
     - Assign target (even distribution)
     - Calculate damage (base + bonus - armor)
     - Apply mechanics: splash, trample, extra projectiles,
       dodge shield, bleed, block, bonus per kill, hp regen,
       charge projectiles, accuracy, siege area damage
     - Apply damage to target (and splash neighbors)
   Remove dead units, check dismount/transform triggers

4. Winner = side with survivors (or draw if timeout)
```

### 18 Combat Mechanics (all data-driven)

| Mechanic | Field | Example |
|----------|-------|---------|
| Siege splash | `is_siege_projectile`, `splash_radius` | Mangonel, Onager |
| Trample (percent) | `trample_percent`, `trample_radius` | War Elephant (50%) |
| Trample (flat) | `trample_flat_damage`, `trample_radius` | Cataphract + Logistica (5 dmg) |
| Ignore melee armor | `ignores_melee_armor` | Leitis |
| Ignore pierce armor | `ignores_pierce_armor` | Leitis |
| Extra projectiles | `extra_projectiles`, `extra_projectile_attacks_json` | Chu Ko Nu (2-4 extra) |
| Charge projectiles | `charge_projectile_count/speed/attacks_json` | Fire Lancer |
| First-attack extra | `first_attack_extra_projectiles` | (reserved) |
| Splash on hit | `splash_on_hit_radius` | (reserved) |
| Dodge shield | `dodge_shield_max`, `dodge_shield_recharge` | Shrivamsha Rider |
| Bleed | `bleed_dps`, `bleed_duration` | (reserved) |
| Block first melee | `block_first_melee` | (reserved) |
| Attack bonus per kill | `attack_bonus_per_kill` | (reserved) |
| HP regeneration | `hp_regen` | Berserk (40 HP/min) |
| HP transform | `hp_transform_threshold` | (reserved) |
| Dismount on death | `dismount_*` fields | Konnik |
| Bonus damage reduction | `bonus_damage_reduction` | Sicilian units |
| Accuracy | `accuracy` | Cavalry Archer (50-100%) |

### Performance

- ~1.3ms per `simulate_battle()` call
- Full matchup advisor analysis endpoint: ~1.5s (phases 1-3), ~4s (phase 4)
- Pure Python lists (no numpy dependency)
- `prepare_combat_unit()` parses JSON once per unit, not per simulation
- Pair cache in matchup advisor avoids redundant sims (keyed by `id()` of unit dicts)

| Data | Stored in | Generated by |
|------|-----------|-------------|

| Projectile speed/attacks | `extracted_data/units.json` (resolved from projectile units) | `extract_units.py` |
| Charge mechanics | `extracted_data/units.json` (charge_type, charge_projectile_unit) | `extract_units.py` |
| HP regeneration | `extracted_data/units.json` (rear_attack_modifier) | `extract_units.py` |

### Computed Data (per-civ fully-upgraded)

| Data | Stored in | Generated by |
|------|-----------|-------------|
| Final stats after all techs | `aoe2_units.db` → `unit_stats` | `generate_database.py` |
| Stat chain (snapshot per tech) | `aoe2_reference.db` → `ref_stat_chain` | `generate_reference.py` |
| Applied techs list | `aoe2_reference.db` → `ref_techs_applied` | `generate_reference.py` |
| Special combat properties | `aoe2_reference.db` → `ref_special_effects` | `generate_reference.py` |
| Projectile breakdown | `aoe2_reference.db` → `ref_projectiles` | `generate_reference.py` |

### Hardcoded Overrides (in `config.py`)

| Data | Location | Why hardcoded |
|------|----------|--------------|
| Ignores armor (Leitis) | `UNIQUE_COMBAT_PROPERTIES` | No dat file source |
| Wootz Steel (Dravidians) | `CIV_COMBAT_PROPERTIES` | Attribute 63 is multi-purpose |
| Logistica trample (Cataphract) | `CIV_COMBAT_PROPERTIES` | blast_damage=-5.0 in dat (no trample) |
| Siege projectile flag | `COMBAT_PROPERTIES` | Supplemental to extracted data |
| Unit line definitions | `CASTLE_UNITS`, `IMPERIAL_UNITS`, `UNIQUE_UNITS` | Game structure not in dat |

### Battle Scores (pre-computed)

| Data | Stored in | Generated by |
|------|-----------|-------------|
| Round-robin scores (per unit line) | `battle_scores.json` | `compute_battle_scores.py` |
| Benchmark scores (vs reference units) | `battle_scores.json` | `compute_battle_scores.py` |

## Key Design Decisions

1. **Two databases**: Main DB for fast flat queries (unit browser, simulator), Reference DB for audit trail (tech chains, stat progression). Separation keeps the main DB simple while reference DB supports deep inspection.

2. **Pre-computed battle scores**: The rankings page would be too slow to compute on the fly (~1.3ms × 13 civs × 13 civs × 16 unit lines = minutes). `compute_battle_scores.py` runs once after DB regeneration and saves results to JSON.

3. **Backend simulation has no positions**: Positions would make simulations 100x slower and add complexity without improving accuracy for ranking purposes. The frontend simulation has full 2D positions for visual appeal.

4. **Combat properties are layered**: Extracted data (correct for most units) → hardcoded overrides (for edge cases like Leitis). This minimizes hardcoding while handling dat file quirks.

5. **Progressive loading in Matchup Advisor**: Phases 1-3 are fast (comparison logic, ~1.5s). Phase 4 is slow (25 army simulations per age × 5 splits each = 625 sims). Splitting into two endpoints lets the UI show analysis immediately.

6. **Pair cache with `id()` keys**: The matchup advisor runs many unit-pair simulations that overlap across phases. Caching by Python object `id()` (not slug) correctly handles same-slug units from different civs (e.g., both civs' Heavy Cavalry Archer having slug `heavy_cav_archer`).
