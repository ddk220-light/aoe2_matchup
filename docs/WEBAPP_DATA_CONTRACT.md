# Webapp Data Contract (Keep UI Unchanged)

Goal: regenerate data from scratch while keeping current webapp/template code untouched.

## 1. Required databases and tables

The webapp reads both:
- `webapp/aoe2_units.db`
- `webapp/aoe2_reference.db`

### `aoe2_units.db` (minimum required)

Tables used directly:
- `civilizations(id, name)`
- `ages(id, name)`
- `units(id, slug, display_name, age_id, unit_type, civ_id)`
- `unit_stats(...)` (large combat + cost + mechanic column set)
- `armor_classes(id, name)`

`unit_stats` must include all simulator-read columns, especially:
- core: `hp, attack, attack_range, attack_speed, attack_delay, melee_armor, pierce_armor, movement_speed`
- json: `attacks_json, armors_json`
- mechanics: `min_attack_range, is_siege_projectile, splash_radius, projectile_speed, ignores_*`
- projectile variants: `extra_projectiles, extra_projectile_attacks_json, charge_projectile_*`
- status/unique: `dodge_shield_*, bleed_*, block_first_melee, attack_bonus_per_kill, pass_through_percent`
- advanced: `transform_*`, `dismount_*`, `armor_strip_per_hit`, `charge_attack_melee`, `charge_recharge_time`, `bonus_hp_nearby`, etc.

### `aoe2_reference.db` (minimum required)

Tables used directly:
- `ref_units`
- `ref_techs_applied`
- `ref_stat_chain`
- `ref_special_effects`
- `ref_projectiles`

The simulator endpoint (`/api/ref/combat-unit/...`) builds response data by combining all five.

## 2. API contracts the frontend depends on

### `GET /api/ref/civ/<civ_name>`
Used by selection UIs and civ detail screens.

Expected payload shape (critical fields):
- `units_by_age.Castle[]` and `units_by_age.Imperial[]`
- each unit includes: `unit_name`, `unit_slug`, `unit_type`, `stat_chain`, `projectiles`, `special_effects`, base/final stats, costs

### `GET /api/ref/combat-unit/<civ_name>/<unit_slug>`
Used by simulator and debug formulas.

Must include:
- identity: `name`, `civ`, `slug`, `unit_name`
- core combat fields and json strings `attacks_json`, `armors_json`
- mechanic fields listed above
- `stat_chain` array with `attacks_json` and `armors_json` per step
- costing: `total_cost`, `upgrade_cost_food`, `upgrade_cost_wood`, `upgrade_cost_gold`
- rendering hint: `outline_size`

### `GET /api/armor-classes`
Must return id->name map used by debug damage breakdown.

## 3. JSON encoding rules you must preserve

- `attacks_json` and `armors_json` are JSON objects with stringified class IDs as keys.
- Frontend parses these strings directly.
- Missing JSON should be `null` or empty object string consistently; avoid invalid JSON.

## 4. Slug and civ routing rules

The URL path key for combat lookup is slug-based.

Therefore:
- `ref_units.unit_slug` must exactly match unit slug emitted in `/api/ref/civ` and consumed by `/api/ref/combat-unit`.
- Unique unit slugs must include civ suffix convention used by UI pickers.

## 5. Icon and SVG expectations

### Unit/building visuals in simulator and civ pages

The UI primarily uses PNG icon CDNs, not backend-generated SVG:
- primary: `https://raw.githubusercontent.com/qwyt/aoe2-icon-resources/master/objects/<id>.png`
- fallback: `https://aoe2techtree.net/img/Units/<id>.png`

Icon resolution path:
- frontend maps `unit_name` -> icon ID via `NAME_TO_ICON` object
- then builds PNG URL

So “getting SVG” for unit icons is not required by current simulator flow.

### Where SVG is actually used

- inline UI icons and placeholder assets in templates (`<svg ...>` or `data:image/svg+xml,...`)
- these are hardcoded presentation assets, not extracted from dat

If you want true SVG unit assets, that is a new asset pipeline and would require frontend changes.

## 6. Battle scores and unit rankings

### Pre-computed files

- `webapp/battle_scores.json` — round-robin and benchmark scores for non-infantry lines
- `webapp/aoe2_reference.db` `battle_scores` table — infantry role-based scores

Both are generated by `webapp/compute_battle_scores.py` (run after regenerating reference DB).

### `battle_scores` table schema

```
battle_scores(line_slug, age, civ_name, unit_slug, score_type, score_value)
```

### Infantry scoring system

All infantry lines (militia, spear, shock_infantry) are pooled together and scored against benchmark opponents. Raw sub-scores are HP%-based (-100 to +100) or survival-time-based, then derived scores are normalized 0–100 across all infantry units.

**Raw sub-scores** (stored in DB, shown in hover cards):
- `vs_skirm`, `vs_halb`, `vs_hussar`, `vs_champ` — 3K resource fights vs Spanish units (-100..+100)
- `vs_paladin_tank`, `vs_champ_tank` — 30v30 survival time (0..~8, higher = lasted longer)
- `vs_arb_raid` — 30 infantry vs 40 Chinese Arbs (-100..+100)
- `vs_paladin`, `vs_hussar_cav` — 3K resource fights (-100..+100)

**Derived scores** (normalized 0–100, best=100, worst=0 across all infantry):
- `melee_power` = avg(vs_skirm, vs_halb, vs_hussar, vs_champ)
- `meat_shield` (Tankiness) = avg(vs_paladin_tank, vs_champ_tank)
- `raid` = vs_arb_raid
- `anti_cav` = avg(vs_paladin, vs_hussar_cav)
- `militia_value` (Score) = 0.5×melee_power + 0.3×meat_shield + 0.1×raid + 0.1×anti_cav

### Non-infantry scoring (battle_scores.json)

Round-robin scores (`score_30v30`, `score_3k`, `score_5k`) are average HP% across all opponents in the line (-100..+100). Benchmark scores (`vs_champ`, `vs_paladin`, `vs_arb`, `pop_vs_*`) are HP% against fixed opponents.

Frontend computes composite scores:
- **PES** (Pop Efficient Score) = weighted avg of 30v30 round-robin + pop benchmarks
- **RES** (Resource Efficient Score) = weighted avg of resource-based battles

### Regeneration

```bash
python3 webapp/compute_battle_scores.py          # incremental (cached)
python3 webapp/compute_battle_scores.py --full    # force full recompute
```

## 7. Data-only rebuild checklist

To keep webapp unchanged, verify all of these:
1. `/api/ref/civ/<valid civ>` returns Castle+Imperial unit arrays with `unit_slug` and `unit_name` populated.
2. `/api/ref/combat-unit/<civ>/<slug>` returns all mechanic fields the JS `BattleUnit` constructor reads.
3. `attacks_json`/`armors_json` parse cleanly in browser.
4. `outline_size` exists (defaulting to `0.2` is acceptable).
5. `armor_classes` table is populated.
6. unique unit slugs/civ suffixes match UI expectations.
7. costs and upgrade costs are present for resource-based army size calculations.

## 8. Recommended validation script set

After regenerating data, run these checks:
- query counts per civ in `ref_units` and `unit_stats`.
- sample API calls for 5 standard + 5 unique unit slugs.
- simulate page smoke test:
  - load civ lists
  - select units
  - start battle
  - debug panel renders formula (needs `stat_chain` + armor classes)
