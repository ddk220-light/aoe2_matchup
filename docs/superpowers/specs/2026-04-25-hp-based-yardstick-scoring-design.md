# HP-Based Yardstick Scoring & Sim Speedups

**Date:** 2026-04-25
**Status:** Design approved вҖ” awaiting implementation plan

## Problem

Current unit ranking scores are derived from `matchup_combos_real.db` by counting how often a unit is the rank-1 recommendation. This produces wrong scores for units that are good in a role but never the chosen recommendation:

- Aztec Elite Jaguar Warrior: `anti_trash = 0` because Aztecs always pick Eagle Warrior or Champion vs trash opponents, never Jaguar вҖ” even though Jaguar performs fine against Halberdier/Skirm.
- Any unit shadowed by a stronger sibling in its civ's tech tree gets undercounted in roles where the sibling is preferred.

The matchup DB only stores the winner per battle; it does not store HP%, game time, survivors, or any margin-of-victory data. We must re-simulate to capture richer outcomes.

## Goals

1. Replace recommendation-frequency scoring with **direct HP-margin scoring** for unit rankings.
2. Capture richer per-battle data (HP%, game time, survivors, resources lost, end reason) so future scoring tweaks don't require re-simulating.
3. Build a **fast targeted batch** (yardstick fights) for iteration, separate from the full overnight matchup batch.
4. Speed up `simulation_real.py` so re-runs are tolerable.

## Non-Goals

- Replacing the matchup_advisor recommendations. The full `matchup_combos_real.db` flow stays as it is for that.
- Castle Age scoring. Imperial-only, matching the current matchup DB.

Both runners use the same `BattleOutcome` schema. Two scoring scripts both write to `battle_scores`, each owning its own (line, civ, unit, score_type) keys вҖ” yardstick scores take priority for `general_combat` / `anti_archer` / `anti_cav` / `anti_trash` / composites; matchup-derived flow keeps writing other score types.

`matchup_combos_real.db` gets a non-destructive schema migration: new outcome columns added as nullable. Existing rows keep their winner-only data; rows re-generated by the next nightly batch fill in the new columns.

## Components

### 1. `simulation_real.py` вҖ” extended to return `BattleOutcome`

```python
@dataclass
class BattleOutcome:
    winner: int                       # 1, 2, 0 (draw)
    end_reason: str                   # "eliminated" | "decisive_lead" | "time_cap"
    game_time_s: float
    team1_hp_pct: float               # remaining HP / starting HP, 0..1
    team2_hp_pct: float
    team1_survivors: int
    team2_survivors: int
    team1_resources_lost: int         # food+wood+gold of dead units
    team2_resources_lost: int
    team1_start_count: int
    team2_start_count: int
```

`simulate_real_battle()` returns this dataclass. Backwards-compat wrappers for old `(winner, *_)` tuple consumers stay in place during migration.

### 2. Sim speed optimizations (pure-Python, no new dependencies)

1. `__slots__` on `BattleUnit` and `Projectile`
2. Squared-distance comparisons everywhere only inequality is needed
3. Local-variable aliasing of hot-loop attributes
4. Grid cell size auto-tuned per sim from `max(unit_radius * 4, max_attack_range)`
5. Skip projectile collision passes when no ranged units present
6. Find-nearest-enemy throttled to every N ticks for distant units
7. Maintain a single `alive_units` list, remove dead in-place
8. **Unit fingerprint dedup**: canonical fingerprint per instantiated unit (final stats + cost + special properties + bonus damage table). Memoize sim outcomes keyed by `(my_fingerprint, opp_fingerprint, my_count, opp_count, scale, seed)`. Cache lives in worker process. Cached outcomes are written to every civ's row that shares the fingerprint вҖ” DB row count unchanged, sim count drops 2вҖ“4Г—.

Combined target: 3вҖ“5Г— faster on the yardstick batch.

Each optimization independently verifiable against `compare_sims.py` вҖ” winners must still match on the 6 canonical pairings, HP% within ~2pp.

### 3. `run_yardstick_battles.py` (new)

Inputs:
- `civ_power_units.json` вҖ” per civ list of "power units" (units that show up in the rankings UI).
- Hard-coded yardstick set: Champion, Paladin, Arbalester, Halberdier, Elite Skirmisher, Hussar.
- Yardsticks instantiated from a single canonical civ each (e.g., Vikings Champion, Britons Halberdier) so every civ's yardstick lookup is deterministic.

Per (civ, my_unit, yardstick, scale) вҖ” scales = `30v30`, `3k`:
1. Run sim with seed_0.
2. If `abs(score_0) <= 10`, run with seed_1 and seed_2; aggregate the 3 outcomes (mean of HP%, game_time, survivors, resources_lost; majority winner).
3. Persist averaged BattleOutcome + `runs_count` (1 or 3) + `score_stddev`.

Multiprocessing pool, fingerprint cache per worker, resume support (skip rows already in DB), --reset flag for full rebuild.

Estimated ~50 civs Г— ~10 power units Г— 6 yardsticks Г— 2 scales = ~6,000 row-targets. With dedup, real sim count is much lower. Wall-clock: ~15вҖ“25 min on 8 workers.

### 4. `yardstick_battles.db` (new)

```sql
CREATE TABLE yardstick_battles (
    id INTEGER PRIMARY KEY,
    civ TEXT NOT NULL,
    my_unit_slug TEXT NOT NULL,
    yardstick_slug TEXT NOT NULL,            -- champion | paladin | arbalester | halberdier | imp_elite_skirm | hussar
    scale TEXT NOT NULL,                     -- '30v30' | '3k'
    my_count INTEGER NOT NULL,
    opp_count INTEGER NOT NULL,

    -- BattleOutcome (averaged across runs)
    winner INTEGER NOT NULL,
    end_reason TEXT NOT NULL,
    game_time_s REAL NOT NULL,
    team1_hp_pct REAL NOT NULL,
    team2_hp_pct REAL NOT NULL,
    team1_survivors INTEGER NOT NULL,
    team2_survivors INTEGER NOT NULL,
    team1_resources_lost INTEGER NOT NULL,
    team2_resources_lost INTEGER NOT NULL,
    team1_start_count INTEGER NOT NULL,
    team2_start_count INTEGER NOT NULL,

    -- Repetition tracking
    runs_count INTEGER NOT NULL,             -- 1 or 3
    score_stddev REAL,                       -- NULL when runs_count=1

    UNIQUE(civ, my_unit_slug, yardstick_slug, scale)
);
```

`generate_matchup_db_real.py` updated to populate these fields going forward. Old rows keep NULL outcome fields until the next nightly run regenerates them.

### 6. `derive_scores_from_yardsticks.py` (new)

Reads `yardstick_battles.db`, computes scores, writes to `battle_scores`.

Per-matchup signed score:
```python
signed_score = 100 * (winner_hp_pct - loser_hp_pct)
if winner == team2: signed_score = -signed_score
```

Per-role aggregation, averaged across the 2 scales (30v30 and 3k):
```
anti_archer    = avg(score vs Arbalester)
anti_cav       = avg(score vs Paladin)
anti_trash     = avg(score vs Halberdier, Elite Skirm, Hussar)
general_combat = avg(score vs Champion, Paladin, Arbalester)
```

Composites (unchanged from current scoring):
```
militia_value         = 0.75 * GC + 0.10 * AC + 0.15 * AT
ranged_effectiveness  = (0.70 * GC + 0.30 * AA) * (final_speed * (final_range + 1))
stable_effectiveness  = (0.70 * GC + 0.30 * AC) * final_speed
```

Pool-normalize each score across the entire pool (infantry / ranged / stable), min-max вҶ’ 0..100. Same `POOL_OF_LINE` map as `derive_battle_scores_from_matchups.py`.

Also persist per-yardstick raw scores (`gc_30v30_vs_champ`, `at_3k_vs_halb`, etc.) вҖ” useful for unit detail page tooltips, mirroring the existing `_raw` benchmark scores.

Targeted DELETE: only the (line, civ, unit, score_type) rows being replaced. Non-power units retain whatever they had.

Backup before write: `webapp/backups/battle_scores_pre_yardstick_<timestamp>.json`. `--restore` flag.

### 7. `derive_battle_scores_from_matchups.py` updates

Mostly unchanged, but coordinated with the new yardstick deriver:

- The 7 score types currently owned by this script (`general_combat`, `anti_archer`, `anti_cav`, `anti_trash`, `militia_value`, `ranged_effectiveness`, `stable_effectiveness`) move to **yardstick deriver ownership**.
- Matchup deriver continues to own anything we choose to leave it (e.g., a future `recommendation_frequency` score, if useful).
- Runs after the yardstick deriver in the standard pipeline.

For now, simplest: stop writing those 7 score types from the matchup deriver entirely. Yardstick deriver becomes the single writer.

## Pipeline

Standard refresh after sim changes:

```bash
# 1. Edit simulation_real.py (add an ability, fix a bug, tune speed)
# 2. Validate: 6 canonical pairings still produce same winners
cd webapp && python3 compare_sims.py

# 3. Re-run yardstick batch (~15-25 min)
cd webapp && python3 run_yardstick_battles.py

# 4. Re-derive ranking scores (seconds)
cd webapp && python3 derive_scores_from_yardsticks.py

# 5. (Optional, overnight) re-run full matchup batch for advisor
cd webapp && python3 generate_matchup_db_real.py --reset
cd webapp && python3 derive_battle_scores_from_matchups.py    # any remaining score types
```
